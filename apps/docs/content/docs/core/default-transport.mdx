# DefaultChatTransport

`DefaultChatTransport` 는 **canonical 와이어 포맷**(`axe-wire/1`) 을 말하는
서버에 그대로 연결할 수 있는 HTTP+SSE 전송 계층입니다. 스키마 매핑, 필드
리네이밍, 감싸기 규칙이 전혀 없습니다 — 서버가 규격대로 내려주기만 하면 됩니다.

```ts
import { DefaultChatTransport } from '@axe-ai-sdk/core'

const transport = new DefaultChatTransport({
  api: '/api/chat',
  headers: () => ({ Authorization: `Bearer ${getToken()}` }),
  prepareBody: (request, state) => ({
    messages: request.messages,
    conversationId: state.conversationId,
    temperature: 0.7,
  }),
})
```

## 옵션

| 옵션 | 타입 | 설명 |
| --- | --- | --- |
| `api` | `string` | POST 할 엔드포인트 URL |
| `headers` | `Record<string,string>` 또는 `() => ...` | 정적 혹은 매 요청 resolver. 토큰 주입에 사용 |
| `prepareBody` | `(request, state) => object` | POST body 빌더. 기본값: `{ messages, conversationId? }` |
| `fetch` | `typeof fetch` | SSR/테스트용 커스텀 fetch |

`metadata.conversationId` 는 자동으로 추적됩니다. 서버가 시작 이벤트에
`conversationId` 를 실어주면 다음 요청 body 에 자동 포함됩니다.

## axe-wire/1 — canonical SSE 포맷

규약은 단 두 줄입니다.

1. **`event:` 이름이 `StreamPart.type` 과 동일** — `text-delta`,
   `thinking-step`, `tool-call`, `tool-result`, `citation`, `message-start`,
   `metadata`, `error`, `finish` 중 하나.
2. **`data:` 는 JSON 객체**. `StreamPart` 의 나머지 필드를 그대로 담습니다.

```
event: message-start
data: {"messageId":"msg_01abc","role":"assistant"}

event: metadata
data: {"data":{"conversationId":"conv_123"}}

event: thinking-step
data: {"step":{"agent":"planner","status":"running","thought":"Parsing intent"}}

event: thinking-step
data: {"step":{"agent":"planner","status":"complete"}}

event: citation
data: {"citation":{"id":"doc-1","title":"Getting started","url":"https://example.com/..."}}

event: text-delta
data: {"delta":"안"}

event: text-delta
data: {"delta":"녕"}

event: finish
data: {"reason":"stop"}
```

이 모양이면 `DefaultChatTransport` 는 `{ type: event, ...JSON.parse(data) }`
로 읽어 그대로 yield 합니다. 추가 설정은 없습니다.

<Callout type="info">
  편의상 `data: [DONE]` 는 `{ type: 'finish', reason: 'stop' }` 의 별칭으로
  처리됩니다. 하지만 canonical 포맷에서는 `event: finish` 를 권장합니다.
</Callout>

### 알려지지 않은 이벤트

`StreamPart.type` 이 아닌 `event:` 이름은 **조용히 무시**됩니다. 상태 기계를
깨뜨리지 않으며, `onSSE()` 구독자에게는 여전히 raw 이벤트가 전달됩니다.
덕분에 `<SSEDebugPanel>` 에서 `partCount: 0` 으로 표시되어 **규격을 벗어난
이벤트를 즉시 찾을 수 있습니다**.

## 인증 — `headers` resolver 로 토큰 주입

토큰은 어디에 저장하느냐(localStorage / 쿠키 / context / ref) 가 앱마다 다르기
때문에, `DefaultChatTransport` 는 auth 개념을 직접 갖지 않고 **헤더 resolver**
하나만 받습니다. Resolver 는 **매 요청마다 호출**되므로 세션 중간에 토큰이
바뀌어도 다음 요청부터 자동 반영됩니다.

### 쿠키에서 bearer 토큰

운영 환경에서 로그인 플로우가 쿠키에 토큰을 세팅하는 경우, `bearerFromCookie`
유틸을 그대로 꽂으면 끝입니다.

```ts
import { DefaultChatTransport, bearerFromCookie } from '@axe-ai-sdk/core'

new DefaultChatTransport({
  api: '/api/chat',
  headers: bearerFromCookie('gateway_token'),
})
```

- 쿠키가 없으면 빈 객체를 리턴 — `Authorization` 헤더 없이 요청이 나갑니다.
- URL-encoded 이름/값을 자동 디코드합니다.
- SSR/Node/테스트 환경에서는 `document` 가 없으므로 조용히 `null` → 빈 객체.

### 여러 헤더 조합

`bearerFromCookie` 는 단순한 resolver 팩토리일 뿐이니 다른 헤더와 자유롭게
합칠 수 있습니다.

```ts
import { bearerFromCookie } from '@axe-ai-sdk/core'

const auth = bearerFromCookie('gateway_token')

new DefaultChatTransport({
  api: '/api/chat',
  headers: () => ({
    ...auth(),
    'X-Tenant': getTenant(),
    'X-Trace-Id': crypto.randomUUID(),
  }),
})
```

### 환경변수 폴백

로컬 개발에서는 `.env.local` 에 토큰을 박아두고 운영에서는 쿠키를 쓰는 흔한
패턴은 다음과 같이 처리합니다.

```ts
import { getCookie } from '@axe-ai-sdk/core'

new DefaultChatTransport({
  api: '/api/chat',
  headers: () => {
    const token =
      import.meta.env.VITE_GATEWAY_TOKEN ?? getCookie('gateway_token')
    const out: Record<string, string> = {}
    if (token) out.Authorization = `Bearer ${token}`
    return out
  },
})
```

### HttpOnly 쿠키

JS 로 읽을 수 없는 HttpOnly 쿠키를 쓰는 경우에는 `getCookie` 가 쓸모없습니다.
대신 fetch 에 `credentials: 'include'` 를 줘서 브라우저가 쿠키를 자동으로
실어보내게 해야 합니다. `DefaultChatTransport` 는 아직 `credentials` 옵션을
직접 받지 않으므로, 이 경우엔 `ChatTransport.send` 를 직접 구현해 fetch
옵션을 세밀하게 제어하는 편이 맞습니다.

<Callout type="warning">
  SDK 는 **401 자동 리프레시**, **localStorage 헬퍼**, **OAuth 플로우** 같은
  auth 기능을 의도적으로 포함하지 않습니다. 앱마다 정책이 너무 달라서 SDK 에
  들어오면 오히려 디버깅 지옥이 되기 때문입니다. 리프레시 로직은 앱 레이어
  (예: fetch wrapper, route guard, 또는 백그라운드 타이머) 에서 처리하세요.
</Callout>

## Raw 이벤트 구독 — `onSSE()`

디버깅, 시각화, 혹은 `<SSEDebugPanel>` 연결용으로 raw 이벤트 스트림을 구독할
수 있습니다.

```ts
const unsub = transport.onSSE((e) => {
  console.log(e.event, e.data, e.parts)
})
// 나중에: unsub()
```

React 에서는 `useSSELog(transport)` 나 `<SSEDebugPanel transport={...} />` 를
쓰는 편이 더 간단합니다.

## 규격이 다른 백엔드

기존 OpenAI / Anthropic / 레거시 게이트웨이 등 **canonical 포맷이 아닌** 서버에
붙일 때는 두 가지 선택지가 있습니다.

1. **서버 측에 얇은 어댑터 레이어** — Next.js route handler, Cloudflare
   Worker, BFF 등에서 한 번만 canonical 포맷으로 번역해 내려주면 클라이언트는
   아무 설정 없이 동작합니다. 가장 깔끔한 권장 경로입니다.
2. **`ChatTransport.send` 직접 구현** — `parseSSE` 나 원하는 파서를 써서
   서버 이벤트를 `StreamPart` 로 매핑한 뒤 yield. `examples/gateway/src/mock-transport.ts`
   가 완전한 예시입니다.

<Callout type="warning">
  예전 버전의 `SSESchema` / `SSERule` / `wrap` / `fields` 옵션은 제거됐습니다.
  스키마 매핑이 필요한 상황은 대부분 "내 서버가 canonical 포맷이 아님" 을
  드러내는 냄새이고, 규격을 서버로 밀어넣거나 직접 `ChatTransport` 를
  구현하는 편이 더 유지보수성이 좋다고 판단했기 때문입니다.
</Callout>
